<decisions>
## Implementation Decisions

### Content Authoring
- AI-generate all ~30 documents from existing sources: `explanations.js` (14 existing topics) and Docusaurus `.mdx` pages
- Do NOT use internal spec files as source material — keep KB docs user-facing only
- All 6 new explainer topics in scope: harness architecture, workflow composition, model selection guide, Atlas Vector Search setup, vai vs DIY RAG, MCP server deep dive
- Documents live in CLI repo only at `src/kb/corpus/` — no cross-repo Docusaurus sync for v1.7

### Manifest Schema
- SHA-256 checksums of file content (Node's built-in `crypto.createHash('sha256')`)
- Full spec schema: id, path, url, title, type, section, difficulty, estimatedChunks, checksum
- Include placeholder `url` fields pointing to `docs.vaicli.com/kb/...` even though they won't resolve yet — keeps schema compatible with future Docusaurus integration
- Manifest auto-generated by a Node script that scans `src/kb/corpus/`, reads front matter, computes checksums
- Top-level manifest fields: version, generatedAt, chunkStrategy, chunkSize, chunkOverlap, embeddingModel, documents[]

### Bundling Strategy
- Release verification: standalone `scripts/verify-kb-manifest.js` that checks manifest version matches package.json version
- npm package inclusion: Claude decides based on existing package.json files config

### Document Structure
- YAML front matter in each `.md` file: title, type, section, difficulty — manifest generator reads it
- Uniform ~500 words per document across all categories
- Chunk-optimized structure: each H2 section targets ~512 tokens for natural chunk boundaries
- Conversational tone — friendly, approachable, like a senior engineer explaining to a colleague

### Claude's Discretion
- File layout: whether seeding logic lives in `src/kb/` or `src/lib/kb.js` (follow existing project conventions)
- npm package.json files array: verify and update as needed
- Exact set of H2 sections per document type
- Front matter field names (match manifest schema where possible)

</decisions>
