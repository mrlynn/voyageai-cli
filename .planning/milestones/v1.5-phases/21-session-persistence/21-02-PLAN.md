  <name>Task 1: Create SessionSummaryStore for vai_session_summaries</name>
  <files>src/lib/session-summary-store.js, test/lib/session-summary-store.test.js</files>
  <behavior>
    - SessionSummaryStore constructor accepts { db }
    - `storeSummary({ sessionId, summary, embedding })` stores a document with sessionId, summary (string), embedding (float array), createdAt, updatedAt
    - `getSummary(sessionId)` retrieves the summary for a session
    - `ensureIndexes()` creates an index on { sessionId: 1 } (unique) -- the Atlas Vector Search index on `embedding` field is documented but created manually by the user (Atlas Search indexes cannot be created via the MongoDB driver)
    - All operations have try/catch with non-fatal fallback (returns null on failure)
    - `deleteSummary(sessionId)` removes a session's summary
  </behavior>
  <action>
    Create `src/lib/session-summary-store.js` as CommonJS module ('use strict').

    The SessionSummaryStore class:
    1. Constructor stores `this._db = db`, `this._col = null`, `this._client = null`, `this._indexesEnsured = false`
    2. `async _connect()`: lazy connection using getMongoCollection(db, 'vai_session_summaries'). Stores client and collection. On error, logs nothing, returns false.
    3. `async ensureIndexes()`: creates `{ sessionId: 1 }` unique index on the collection. Non-fatal.
    4. `async storeSummary({ sessionId, summary, embedding })`: upsert by sessionId. Document shape: `{ sessionId, summary, embedding, createdAt (only on insert), updatedAt }`. Use `updateOne` with `$set` and `$setOnInsert: { createdAt }` plus `upsert: true`.
    5. `async getSummary(sessionId)`: `findOne({ sessionId })`. Returns null on not found or error.
    6. `async deleteSummary(sessionId)`: `deleteOne({ sessionId })`. Returns true/false.
    7. `async close()`: close client if connected.

    Export: `{ SessionSummaryStore }`

    For tests, use mock collection pattern (same as 21-01). Test:
    - storeSummary creates proper document shape with embedding array
    - getSummary returns stored summary
    - deleteSummary removes summary
    - ensureIndexes calls createIndex with correct spec
    - Connection failure returns graceful nulls (no throw)
  </action>

  <action>
    This task wires SessionStore into the existing chat flow and adds session management commands.

    **Changes to src/commands/chat.js:**

    1. Add new CLI options to the chat command registration:
       - `--list` flag: "List recent chat sessions and exit"
       - `--all` flag: "Include archived sessions in --list output"

    2. Add `--list` handler at the top of `runChat()` (before any other setup):
       ```javascript
       if (opts.list) {
         const { SessionStore } = require('../lib/session-store');
         const store = new SessionStore({ db });
         // Query vai_sessions, filter by lifecycleState != 'archived' unless --all
         // Display table: Session ID (first 8 chars) | First message (truncated 50 chars) | Turns | Last active | Status
         // Use ui helpers for table formatting
         await store.close();
         return;
       }
       ```

    3. Replace the ChatHistory + mongo setup section in `runChat()`:
       - Create SessionStore instance: `const store = new SessionStore({ db })`
       - If `--session <id>` provided:
         - Call `store.getSession(sessionId)` to verify it exists
         - Call `store.transitionLifecycle(sessionId, 'active')` if it was paused
         - Load turns via `store.getLatestTurns(sessionId, maxTurns * 2)`
         - Populate ChatHistory from loaded turns
         - Display "Resumed session {id} ({N} turns loaded)" message
       - If no --session:
         - Call `store.createSession({ model, provider, mode })` to get new session
         - Use returned session._id as the sessionId
       - If `opts.history === false` (--no-history): skip store creation, use ChatHistory in-memory only (existing behavior)
       - Pass `store` to handlePipelineTurn / handleAgentTurn via the context object

    4. After each turn completes (in the 'done' event handler), persist the turn:
       ```javascript
       if (store && !store.isFallbackMode) {
         await store.storeTurn({
           sessionId,
           turnIndex: orchestrator.turnIndex,
           role: 'user',
           content: query,
           tokens: { estimated: estimateTokens(query) },
           timing: { startedAt: turnStart, completedAt: Date.now(), durationMs: Date.now() - turnStart }
         });
         await store.storeTurn({
           sessionId,
           turnIndex: orchestrator.turnIndex,
           role: 'assistant',
           content: fullResponse,
           context: retrievedDocs,
           tokens: metadata.tokens || {},
           timing: { startedAt: turnStart, completedAt: Date.now(), durationMs: Date.now() - turnStart }
         });
         await store.updateSession(sessionId, { updatedAt: new Date() });
       }
       ```

    5. Add `/archive` slash command handler in the existing slash command section of the chat loop:
       - When user types `/archive`: call `store.transitionLifecycle(sessionId, 'archived')`
       - Display "Session archived." and continue prompting (or exit, user's choice)
       - If store is in fallback mode, display "Session archiving not available (no MongoDB connection)"

    6. Add `/sessions` slash command: same as --list but inline during chat. Shows recent sessions.

    7. On chat exit (cleanup section), call `store.close()` if store exists.

    8. If store switches to fallback mode during the session, display a one-time dim message: "MongoDB unavailable — session will not be persisted" (SES-06 UX).

    **Changes to src/lib/history.js:**

    Update ChatHistory to optionally accept a SessionStore instead of raw mongo:
    - Add `store` option to constructor: `{ sessionId, maxTurns, store, mongo }`
    - If `store` is provided, use it for persistence in `addTurn()` and `load()` instead of raw collection access
    - `load()` with store: calls `store.getLatestTurns(sessionId, maxTurns * 2)` and populates turns array
    - `addTurn()` with store: calls `store.storeTurn(...)` after adding to in-memory array
    - Keep backward compatibility: if `mongo` is provided (old way), use existing behavior
    - This is a gradual migration -- both paths work simultaneously

    Do NOT change the ChatHistory.getMessages(), getMessagesWithBudget(), exportMarkdown(), exportJSON() methods -- they operate on the in-memory turns array which is populated either way.
